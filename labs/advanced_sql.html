When to Use Indexes

  • Columns frequently used in WHERE clauses
  • JOIN columns (foreign keys)
  • Columns used in ORDER BY or GROUP BY
  • High-cardinality columns (many distinct values)
  • Large tables where full scans are expensive

When to Avoid Indexes

  • Small tables (sequential scan is fast enough)
  • Columns with low cardinality (e.g., boolean flags)
  • Tables with heavy INSERT/UPDATE/DELETE workloads
  • Columns rarely used in query predicates
  • When queries return a large fraction of rows

Common Index Types

  • B-Tree — default, supports =, <, >, BETWEEN
  • Hash — equality comparisons only (=)
  • GIN — full-text search, arrays, JSONB
  • GiST — geometric data, range types
  • BRIN — very large, naturally ordered tables

Index Complexity

  • Search: O(log n) — follows tree from root to leaf
  • Insert: O(log n) — find leaf + possible splits
  • Full Scan: O(n) — reads every row sequentially
  • Space: O(n) — additional storage for index
  • Splits: propagate upward, keep tree balanced
